Burn-up Chart

New Stories Added

Except for the stories we added in Sprint 4 (32 points in addition to the initial 40 of the project) and worked on Sprint 5 (see here for more details), we added another 41 story points bringing the total up to 113 points.

 

As we perhaps should have expected, the end project has almost 3x as many points as the original. For that reason, we chose to make a burn-up chart to represent the incremental work done above and beyond project requirements.

Sprint Completed Stories

The new stories added and completed during this sprint were:

  • Create database instance
  • Migrate events create to new DB
  • Modify code in new branch to link with database
  • Thoroughly test and fix (many many many) bugs
  • Sort events by date
  • Change wording of past events to “attended”
  •  Add year to events (in addition to month and day)
  • Add controls for non-existing dates (e.g. February 30), taking leap-years into account
  • Add funny 404 landing page

Remaining Stories

 

As with all projects, our work is never over (listen here for the mantra of ‘80s kids), hence we have left these stories (21 points) for future sprints:

  • Create mobile site and direct traffic based on device type
  • Improve site security (especially against on XSS attacks)

What did we accomplish? Roadblocks? Feedback?

 

During the last sprint we did not have any roadblocks and we accomplished to go far above and beyond what was required from the project. In terms of feedback, we were very efficient in doing the work, having accumulated a lot of experience from previous sprints. As can be seen from the burn-up chart, we were much more efficient in our work and we felt increasingly confident in adding more and more user stories (first addition was for ~75% additional workload versus project requirements, second addition was another ~100% more).

Final Thoughts

 

Concluding this report, we would like to say that our site is freaking awesome. But don’t take our word for it, here it is!!

  • Add google analytics

  • Add logic to tell apart past events from future (e.g. “attended/attending” where appropriate)
