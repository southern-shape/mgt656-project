We first assessed the difficulty and allocated points to each of the steps described as user stories. We labeled the stories from A to K (in the order they appeared in the brief) and gave them points, which we averaged to have the final estimate. While giving points, we each came up with our own estimates in private, so that we could average independent observations.

 

Our second task was to assess their relative position in the backlog funnel. We completed this through group discussion around what we wanted to achieve, what was required for rapid prototyping, and which stories were dependent on others. We split the first story in 3 parts as we thought that some of its elements were critical to be in the beginning and others were not as pressing; we then rescored. We then sorted our stories in a backlog.

 

Third, we allocated each story to a sprint, such that workload across sprints would be roughly evenly distributed. In one case (story H), we said we would move it to the next sprint as it is quite large and we did not think that it would be meaningful to have it half-completed after the sprint.

How the team will function

 

1.     What roles will you assume in your teams?

 

There will be 3 roles in our 3-person team: product manager, scrum master, and developer.

 

2.     How will you assign those roles?

 

We have decided that Vas will be the scrum master, Karan will be the developer, and Ethan will be the product manager. In our meeting, we discussed capabilities and desired takeaways from the course. Either by providence or design, there was a lot of complementarity in the team, so role allocation was natural and voluntary.

 

3.     What meetings do you plan to have? Please plan on using a variant of Scrum.

 

We want to follow the principles of agile, but since we are not staffed on this project full-time, we agreed that we would have the following meetings:

-        Planning meeting after class every Thursday (after the retrospective meeting, if applicable) for roughly 2 hours at SOM

-        “Daily” meeting, every 2-3 days, agreed upon after every planning meeting and based on workload; meetings will be done either physically or virtually, as necessary

-        Review meeting during lunch every Thursday for 30 minutes to 1 hour at SOM

-        Retrospective meeting after class every Thursday for roughly one hour at SOM

 

4.     How will you decide when to have meetings?

 

On any type of meeting other than “daily,” we already have decided, hence we will only adjust if necessary. In terms of “daily” meetings, we will decide during the planning meeting based on expected workload and need for alignment. As a general direction, for sprints that have a lot of interdependencies, we will need to have more “daily” meetings. We will also will have quick ad-hoc meetings that we will be able to organize through IM software.

 

5.     How long do you expect the project to take?

 

In the first sprint, we spent roughly 2 hours of total time for 5 allocated points. Therefore, our first estimate is 0.4 hours (24 mins) per point. Since we have 35 more points to complete, we need roughly 14 hours of total time. If we assume that we will have up to 20-30% more points from redevelopment and new stories, we will need 14-18 hours of total work, or 4.5-6 hours of work per person for the entire project.

 

6.     How will you divide up the work?

 

We will divide the work during the planning meeting based on assigned roles and specific sprint needs. Up to now, we divided the work into sprints in a high-level, using the “top-level” stories. In the planning sprints, we will take a more granular approach of dividing up the work in “sub-levels” as described in user stories and/or as re-defined by us. After we have a granular view, we will use a “pull” system, whereby every member will volunteer to do a specific piece. The scrum master will be responsible for making sure that work is fairly allocated and will make changes as necessary.

Central Repo Management - GitHub

 

We have agreed on following terms to version control our code. We will be using central repo management using GitHub:

## Everyone works in their own Cloud9 IDE and modifies their local copies of repo

## Once the changes are made and you have performed basic testing of your new features

git status                                              # View the state of the repo, note the files that were changed

git add <some-file>                          # Stage all the files one by one

git commit -m "Your message goes here"

git push origin master

if (you see an ERROR) {

## At this point if you see ERROR: "error: failed to push some refs to", it means local history has diverged from the central repository

              ## We do not want the official commits to be overwritten. To incorporate upstream changes

                git pull --rebase origin master

                

                if (you see a MERGE conflict) {

                        ## While performing rebasing if you see a CONFLICT: "(content): Merge conflict in  <some-file>", you will have to resolve it

                            git status                                               # View the conflict

                                                    

                           ## Edit the file(s) accordingly, make sure you get rid of "=====", "<<<<<", and ">>>>>>". Once changes are made and testing is done

                            git add <some-file>                            # Stage all the files one by one

                            git rebase --continue

                            

                           ## Rebase might prompt again if it finds more conflicts. Follow the same steps as above to resolve all of them.

                }

## After we are done synchronizing (rebasing), we can publish the changes successfully            

                git push origin master

}
